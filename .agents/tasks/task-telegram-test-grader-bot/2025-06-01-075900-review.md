# Telegram test grader bot with OMR scanning

A self-contained Telegram bot that generates printable A4 answer sheets with alignment markers, scans photographed sheets using OpenCV bubble detection with perspective correction, grades them against stored correct answers, and exports ranked results to Excel. The bot targets a 2GB AWS free-tier instance already running two other bots, using OMR over OCR to stay under 120MB resident memory.

Watch for: The OMR pipeline silently degrades to a naive resize when corner markers aren't found, which will produce wrong grades without any user-visible warning (confirmed). Session JSON files have no access control — any user who guesses another user's Telegram ID can read/overwrite their test data (confirmed). The deploy script writes BOT_TOKEN to disk in plaintext with no restrictive file permissions (confirmed).

## High-level view

The conversation flow is driven by two `ConversationHandler` chains (new-test and scan) with JSON files as the persistence layer. Sessions are keyed by Telegram user ID with no isolation between "tests," so starting a new test clobbers the previous one's student results while preserving stale records from the prior grading session.

The OMR pipeline couples directly to the sheet generator's layout constants to compute expected bubble positions after a perspective transform. When the transform can't be applied (corners not detected), the fallback resizes the raw photo to sheet dimensions and applies the same grid — a path that will almost always produce garbage grades because the photo won't align to the expected coordinates.

The sheet generator accepts up to 100 questions but the grid physically overflows the A4 page at ~74 questions. Beyond that, bubbles are drawn off-canvas and the scanner returns zero-fill for those positions.

Deployment writes the bot token to `.env` without restrictive permissions, and `ADMIN_ID` is loaded but never checked — the bot is open to any Telegram user.

## OMR pipeline: perspective correction fallback

When `_detect_corner_markers` returns `None`, the code does:

```python
warped_gray = cv2.resize(gray, (SHEET_WIDTH, SHEET_HEIGHT))
```

This stretches an arbitrary photograph — which may include desk edges, fingers, partial views — to sheet dimensions and then applies the same grid lookup. The fill ratios returned will be effectively random. The function does not set `result["error"]`; it proceeds to grade and return a score. The user sees a number that looks legitimate but isn't.

This should return an error ("Burchaklar topilmadi — qaytadan suratga oling") or apply a confidence heuristic on the fill ratios and reject scans where detected marks are implausibly sparse or uniform.

## Session lifecycle and data integrity

The `/new_test` handler loads the existing session, overwrites `num_questions`, `num_options`, `correct_answers`, and `correct_letters`, but preserves `students` via `session.get("students", [])`. If a teacher scanned 15 students on a 30-question test then starts a new 20-question test, the old 15 student records remain in the session. A subsequent `/results` exports those stale records whose scores no longer correspond to the new answer key.

A `session["students"] = []` on new test creation would be the minimal fix. A more robust approach: store each test under a unique ID and link students to their test.

## Access control gap

`config.py` loads `ADMIN_ID` but no handler checks it. Any Telegram user who discovers the bot's username can invoke all commands, including image processing that consumes CPU and disk I/O on the shared 2GB server. At minimum, wrap handlers in an `if update.effective_user.id != ADMIN_ID` guard or maintain an allowed-users list.

## Deploy script credential handling

```bash
cat > "$INSTALL_DIR/.env" << EOF
BOT_TOKEN=$BOT_TOKEN
ADMIN_ID=$ADMIN_ID
EOF
```

The file inherits the default umask (typically 022), making it readable by all system users. Adding `chmod 600 "$INSTALL_DIR/.env"` after creation is a one-line fix.

## Grid overflow for high question counts

With `num_questions = 100` and `use_two_columns = True`, column 1 holds 50 questions. The Y coordinate of the last question: `GRID_START_Y + 49 * BUBBLE_SPACING_Y = 550 + 49 * 80 = 4470`, exceeding `SHEET_HEIGHT` (3508). Bubbles are drawn off-canvas and the scanner's ROI clips at image boundaries, returning zero fill ratios. The practical limit is ~37 questions per column (~74 total). Either cap `num_questions` at 74 in the bot's validation, or dynamically reduce `BUBBLE_SPACING_Y` based on question count.

## /results reachability during scan

`/results` is registered as a standalone `CommandHandler` outside both conversation handlers. When a user is in `AWAITING_SCAN_PHOTO`, sending `/results` hits the standalone handler but does not terminate the scan conversation. The user gets their Excel file but remains stuck in scan mode until they send `/cancel`. Fix: add `/results` as a fallback or state handler inside `scan_conv`.
