# ArUco marker rewrite for OMR corner detection

The OMR scanning pipeline replaces its hand-rolled black-square corner detector (5 cascading fallback strategies operating on thresholded contours) with OpenCV's built-in ArUco detector (`DICT_4X4_50`, IDs 0-3). The generator now places identifiable ArUco markers at corners and a QR code with test metadata. The perspective transform maps detected marker centers to their known canonical positions rather than stretching to the full sheet boundary. The 3-method consensus voting for bubble detection is preserved unchanged.

**Watch for:** The QR fallback is dead code that cannot produce alignment (confirmed). `_preprocess_image` is dead code, defined but never called (confirmed). The perspective destination points changed from full-sheet corners to marker centers — geometrically more correct, but a behavioral shift that could surface misalignment if constants drift.

## High-level view

The corner detection pipeline dropped from ~200 lines of threshold/contour/fallback logic to a single ArUco detector call plus rotation loop. ArUco IDs provide implicit orientation, making rotation detection simpler and more reliable than the old approach.

The perspective transform destination changed from `[0,0]...[W-1,H-1]` to marker center coordinates `[100,100]...[2380,3408]`. This maps "where the marker was detected" → "where the marker belongs on the canonical sheet." All bubble positions are defined in the same coordinate space, so alignment is consistent. The old approach mapped marker centers to sheet edges, introducing a ~100px systematic offset that happened to be compensated by the old detector finding markers near the edges.

The QR code fallback is structurally present but functionally inert: the code detects QR corners and then executes `pass`. The QR generation on the sheet adds a dependency (`qrcode[pil]`) and complexity without being consumed by the scanner.

## QR fallback is architecturally unsound for alignment

```python
# QR code fallback
qr_pts = _try_qr_fallback(gray)
if qr_pts is not None:
    pass

return None
```

`_try_qr_fallback` returns 4 points from `cv2.QRCodeDetector().detectAndDecode()` — these are corners of the QR code bounding box, not the sheet's alignment corners. Even if wired into a perspective transform, 4 points from a single QR code in the top-right area cannot define a full-page homography. The fallback concept is architecturally unsound for perspective correction. It could work as a metadata recovery mechanism (reading `num_questions`/`num_options` from the encoded JSON) but not as a geometric alignment source. The entire `_try_qr_fallback` function and the `pass` branch should be removed, or repurposed purely for metadata extraction.

## Dead preprocessing function and noisy-photo edge case

The old pipeline called `_preprocess_image(gray)` before marker detection — sharpening, denoising, and CLAHE enhanced the image for contour-based detection. ArUco's internal adaptive thresholding handles most cases, so omitting preprocessing is defensible for clean photos. However, for very noisy or low-contrast phone photos (strong shadows, low light), preprocessing before ArUco detection could improve marker detection rates. The function remains defined at line 45 creating the impression it participates in the pipeline. If it's intentionally excluded from the ArUco path, delete it; if it might help edge cases, call it before `_detect_aruco_markers`.
