# Consensus-based OMR scanner and shared test storage

A two-part rewrite that (1) replaces single-method bubble detection in `omr_scanner.py` with a 3-method consensus voter to fix the 0-score bug caused by printed letter outlines confusing the old relative-threshold approach, and (2) moves `bot.py` from per-user session files to a shared `data/tests/` directory so multiple admins can collaborate on the same test. The OMR pipeline gains an unsharp-mask/denoise/CLAHE preprocessing stage and auto-rotation fallback (90/180/270) when corners aren't found. The bot gains `/tests` and `/use_test` commands for selecting tests.

**Watch for:** Test ID generation has a TOCTOU race between two admins creating tests simultaneously (confirmed). `_load_test` silently returns `{}` on corruption, so downstream code must guard against missing keys — most paths do, but `_process_scan_image` does not (confirmed). The `active_test_id` stored in `context.user_data` is ephemeral and lost on bot restart, with no automatic recovery path communicated to the user (confirmed).

## High-level view

The OMR rewrite uses three detection methods with orthogonal failure modes (relative baseline, global Otsu, morphological opening) feeding into majority voting. Method A's median-baseline subtraction nullifies the printed letter artifacts that caused the original 0-score bug. The fixed thresholds in Method B (0.35) and Method C (0.20) are reasonable for standard printed sheets but may need adjustment for very light pencil marks.

The bot storage migration's main architectural tension is that test selection lives in `context.user_data` (in-memory, per-process), meaning a bot restart during a scan session loses the user's active test link without warning. For a 2GB free-tier server running 3 bots, OOM restarts are plausible.

The `_generate_test_id` function uses a scan-then-create pattern that races under concurrent access. For two admins on a single-process polling bot this is unlikely but not impossible if the bot is later moved to webhook mode.

## 3-method consensus voter

Method A subtracts the median fill ratio across all options for a question, treating it as an estimate of the "empty bubble with printed letter" baseline. This directly targets the root cause: when every bubble has ~0.12 fill from printed outlines, the old code's `MIN_FILL_RATIO = 0.12` threshold flagged all bubbles as marked. After baseline subtraction only the genuinely darkened bubble retains significant fill.

Method B's fixed 0.35 threshold means any bubble with more than 35% fill within its circular mask is considered marked. This will miss light pencil fills that don't cross the 35% boundary — in that scenario the consensus voter relies entirely on Methods A and C agreeing.

One behavioral subtlety in `_determine_answer`: if Method A returns `-1` (ambiguous) and Methods B and C both return `None`, the voter falls through to the `any(r == -1)` check and returns `-1` (uncertain). This means a single-method ambiguity signal is surfaced rather than silently dropped — appropriate for the uncertain-questions workflow in the bot.

## Double preprocessing

`scan_answer_sheet` does:
```python
preprocessed = _preprocess_image(gray)        # sharpen + denoise + CLAHE
corners = _detect_corner_markers(preprocessed)
```

But inside `_detect_corner_markers`:
```python
preprocessed = _preprocess_image(gray)  # called again on the already-preprocessed input
_, binary_otsu = cv2.threshold(preprocessed, ...)
```

The input named `gray` inside `_detect_corner_markers` is actually the already-preprocessed image. The image gets unsharp-mask → denoise → CLAHE → unsharp-mask → denoise → CLAHE. Double-sharpening amplifies noise and can cause ringing artifacts that hurt contour detection. Either remove the internal `_preprocess_image` call or rename the parameter to signal what's expected.

## `_process_scan_image` missing guard

When `_load_test(test_id)` returns `{}` (file deleted or corrupted between scan_command and actual photo processing):

```python
result = scan_answer_sheet(
    tmp_path,
    test_data["num_questions"],  # KeyError on empty dict
    ...
)
```

The `scan_command` handler checks `test_data.get("correct_answers")` before entering scan state, but `_process_scan_image` has no equivalent guard. A file deleted between those two points causes an unhandled `KeyError`.

## Test ID generation race

```python
def _generate_test_id() -> str:
    existing = list(TESTS_DIR.glob("test_*.json"))
    ...
    next_num = max(nums) + 1 if nums else 1
    return f"test_{next_num:03d}"
```

The ID is generated here but the file isn't created until `_save_test` is called later in `receive_answers`. If two admins complete the answers step in the same moment, both get the same ID string, and the second `_save_test` call silently overwrites the first test's data.

## Ephemeral active_test_id

`context.user_data["active_test_id"]` is in-memory only. If the bot process restarts (OOM kill, deploy, crash), all users lose their active test selection. The next `/scan` or `/roster` call gives "Avval test tanlang" with no hint about what happened or how to recover. A one-line persistence (write active_test_id to a user-specific file) or a prompt suggesting `/use_test` after restart would eliminate this confusion.

## Auto-rotation order

The rotation loop tries 90°, 180°, 270° sequentially. In practice 180° rotation (upside-down) is the most common orientation mistake when photographing a sheet. Reordering to `[180, 90, 270]` would find the common case faster and avoid unnecessary OpenCV rotations on the constrained server.
